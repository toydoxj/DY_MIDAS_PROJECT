# Persistent Agent Memory — 공유 매뉴얼

이 문서는 `.claude/agents/reviewers/`, `.claude/agents/authors/` 의 모든 에이전트가 공유하는 영속 메모리 시스템 사용 규칙입니다. 각 에이전트 정의는 본 문서를 참조해 보일러플레이트를 인라인하지 않습니다.

## 1. 메모리 디렉토리

각 에이전트는 자신의 이름과 동일한 디렉토리를 갖습니다:

```
.claude/agent-memory/
  rc-beam/             # reviewers/rc-beam.md 가 사용
  rc-slab/             # reviewers/rc-slab.md
  seismic-cert/        # authors/seismic-cert.md
  design-report/       # authors/design-report.md
```

해당 디렉토리는 사전 생성되어 있습니다. **`mkdir` 또는 존재 확인 없이 직접 `Write`** 합니다.

## 2. 메모리 4 종류

| 종류 | 용도 | 저장 시점 |
|------|------|-----------|
| **user** | 사용자의 역할/선호/지식 | 사용자에 대한 사실을 알게 됐을 때 |
| **feedback** | 작업 방식 교정/검증 | 사용자가 교정 OR 비자명한 접근을 인정했을 때 |
| **project** | 프로젝트 진행 상황/의사결정 | 누가/무엇을/왜/언제 알게 됐을 때 (상대 날짜는 절대 날짜로 변환) |
| **reference** | 외부 시스템 위치 | 외부 자원이 어디에 있는지 알게 됐을 때 |

### feedback / project 본문 구조

```markdown
{규칙 또는 사실}

**Why:** {이유 — 사용자가 준 근거. 이유를 알아야 엣지 케이스를 판단할 수 있음}
**How to apply:** {언제/어디서 적용되는지}
```

## 3. 저장 절차 (2 단계)

**Step 1** — 개별 파일 작성:

```markdown
---
name: {{메모리 이름}}
description: {{한 줄 설명 — 미래 대화에서 관련성을 판단하는 근거}}
type: {{user|feedback|project|reference}}
---

{{메모리 본문}}
```

**Step 2** — `.claude/agent-memory/{agent}/MEMORY.md` 인덱스에 한 줄 추가:

```markdown
- [Title](file.md) — one-line hook
```

규칙:
- `MEMORY.md` 는 인덱스. 본문을 직접 쓰지 않음
- 200줄 이후는 잘림 → 간결 유지
- 의미적으로 정리(주제별), 시간순 X
- 중복 작성 금지 — 기존 메모리 갱신 우선
- 잘못된 메모리는 즉시 갱신 또는 제거

## 4. 저장 금지 사항

다음은 메모리에 저장하지 않습니다(사용자가 명시 요청해도):

- 코드 패턴, 컨벤션, 아키텍처, 파일 경로, 프로젝트 구조 — 현재 코드를 읽으면 알 수 있음
- Git 이력, 최근 변경, 누가 무엇을 했는지 — `git log` / `git blame` 이 정답
- 디버깅 해결책 / 수정 레시피 — 수정은 코드에 있고 맥락은 커밋 메시지에 있음
- `CLAUDE.md` 또는 `.claude/rules/` 에 이미 문서화된 내용
- 일시적 작업 세부사항 — 진행 중 상태, 임시 상태, 현재 대화 컨텍스트

사용자가 PR 목록이나 활동 요약을 저장하라고 요청하면 *놀랍거나 비자명한* 부분이 무엇인지 되묻습니다.

## 5. 회상 절차

- 메모리가 관련 있어 보이거나, 사용자가 이전 대화를 참조하면 접근
- 사용자가 명시적으로 "기억해", "확인해", "리콜" 요청하면 **반드시** 접근
- 사용자가 "메모리 무시", "메모리 사용 금지" 라고 하면: `MEMORY.md` 가 비어 있는 것처럼 동작. 기억된 사실을 적용/인용/비교하지 않음
- 메모리가 오래되었을 수 있음 — 답하기 전에 현재 코드/리소스를 읽어 확인
- 메모리와 현재 정보가 충돌하면 **현재 관찰을 신뢰**하고 메모리를 갱신/삭제

## 6. 메모리 기반 추천 전 검증

특정 함수/파일/플래그를 가리키는 메모리는 *작성 시점에는* 존재했다는 주장입니다. 이름이 바뀌거나 사라졌을 수 있습니다:

- 파일 경로 → 존재 확인
- 함수 / 플래그 → grep 검증
- 사용자가 추천을 바로 실행하려 하면 **반드시** 사전 검증

"메모리에 X 가 있다고 적혀 있다" ≠ "X 가 지금 있다"

저장소 상태 요약(아키텍처 스냅샷 등)은 시점 고정. *최근* 또는 *현재* 상태를 묻는다면 `git log` 또는 코드 직접 읽기.

## 7. 메모리 vs 다른 영속화

- **Plan 사용**: 비자명한 구현 작업 시작 전, 사용자와 합의를 위한 플랜
- **Tasks 사용**: 현재 대화의 작업 단계 추적
- **Memory 사용**: 미래 대화에서 유용한 정보 (현재 대화 한정 정보 X)

## 8. 에이전트별 특수 케이스

### KDS 기준 메모리
KDS 조항이 필요한 항목은 사용자에게 질문 → 응답 즉시 `kds_reference.md` 에 기록 → 이후 재질문 없이 활용. 형식:

```markdown
## [조항번호] 조항명
- 출처: KDS XX XX XX : XXXX, 제X조
- 내용: (사용자 제공 원문 또는 요약)
- 적용: (어느 검토에 사용)
- 확인일: YYYY-MM-DD
```

### 프로젝트 범위 공유
이 메모리는 `.claude/` 하위라 **버전 관리로 팀과 공유**됩니다. 개인 정보가 아닌 프로젝트 맥락만 저장합니다.
